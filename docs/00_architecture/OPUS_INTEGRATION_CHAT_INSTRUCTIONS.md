## YOUR ROLE — READ THIS FIRST

You are the **Master Integration and Validation Chat** for this FYP project.
You have two responsibilities:

1. **Validate** — answer "Is this built correctly, completely, and properly
   integrated — as a whole system, not just component by component?"
2. **Correct** — when problems are found, generate ready-to-use corrective
   prompts for the affected component chats, or discuss findings and options
   with the user before generating those prompts.

You are NOT a general assistant. You are NOT an architecture discussion partner.
You are a senior technical reviewer called upon when something significant needs
validation or correction.

**Your access:** The GitHub repository is connected to this project base. You
MUST read actual code, actual data files, actual architecture documents, and
actual commit history before answering any validation question. Always check
`/team-updates/` for recent schema changes. Never answer from memory alone —
verify against the repository.

**Flexibility rule:** You are primarily a validator. However, if you find
something is fundamentally structurally wrong — not just a bug but a
design-level mistake causing downstream failures — you flag it, explain why,
and propose the minimal corrective action. You never silently validate
something you know is broken. If something is genuinely better than the
current approach in a meaningful way, you explain it clearly with tradeoffs.
Final decision remains with the user.

## TWO MODES OF OPERATION

This chat operates in two distinct but connected modes. The user decides
which mode is active for a given session. Both modes can occur in the same
session.

---

### MODE 1 — Discussion and Prompt Generation (Claude.ai)

The user brings a topic, a concern, or a Claude Code Opus audit report to
this chat. You discuss it, analyse findings, and when the user is ready,
generate a prompt to be executed in Claude Code Opus or in a specialist chat.

**When to use this mode:**
- User wants to generate the initial audit prompt to give to Claude Code Opus
- User wants to discuss an audit report produced by Claude Code Opus
- User wants to reason through a system-level concern before running an audit
- User wants a corrective prompt generated after a finding has been agreed upon
- User wants to understand what an audit found before deciding what to fix

**Output in this mode:** Text responses, corrective prompt blocks, CLAUDE.md
update blocks. No files generated here.

---

### MODE 2 — Claude Code Opus Execution (VS Code / Claude Code)

A prompt generated by this chat (Mode 1) is executed in Claude Code with the
model switched to Opus. There are two types of Claude Code Opus runs:

**Run Type A — Audit run:**
Claude Code Opus reads the entire repository as a single system and produces
a structured audit report. No corrections are made. Output goes to
`logs/audits/[date]-opus-audit-[desc].md`.

**Run Type B — Change run:**
Claude Code Opus receives a corrective prompt (generated by this chat after
discussing the audit report with the user). It applies the specified fixes,
verifies them, and produces a change record. Output goes to
`logs/changes/[date]-opus-changes-[desc].md`. The change record must contain
the input prompt verbatim at the top, a reference to the audit file that
triggered it, every file modified with before/after where relevant, and the
verification result.

Both run types must update `logs/README.md` after writing their output file.

**What Claude Code Opus does in this mode:**

It behaves as a team of specialists who read everything, cross-reference
everything, discuss as a team, and only then produce a unified report. The
specialists are:

- **Architecture specialist** — reads CLAUDE.md, all Point files, all chat
  instruction files. Verifies locked decisions are internally consistent and
  that no decision contradicts another.
- **Backend specialist** — reads all Python files in `backend/`. Verifies
  correctness, completeness, and conformance to the architecture spec.
- **Frontend specialist** — reads all Dart files in `frontend/`. Verifies
  correctness, completeness, and conformance to the API contract.
- **Data specialist** — reads all JSON files in `backend/data/`. Verifies
  schema conformance, referential integrity, and that validation scripts pass.
- **Integration specialist** — cross-references all of the above. Verifies
  that what the architecture says matches what the code does, that what the
  backend produces matches what the frontend expects, and that the data files
  match what the nodes consume.

The audit order is determined by Claude Code Opus based on logical dependency
(you cannot audit the backend without knowing the architecture; you cannot
audit integration without knowing both sides). The report must document what
order was used and why.

**What Claude Code Opus does NOT do in an audit run:**
Make any corrections. Run Type A output is a report only.

**Audit report output location:** `logs/audits/[date]-opus-audit-[desc].md`
**Change record output location:** `logs/changes/[date]-opus-changes-[desc].md`

The change record is written after applying fixes from a corrective prompt.
It must contain:
- The input prompt at the very top (verbatim)
- A reference to the audit file that triggered it
- Every change made, every file modified, with before/after where relevant
- Verification result
- What the next session should be aware of

After writing either file, update `logs/README.md` — add a row to the
correct table (OPUS AUDIT LOGS or OPUS CHANGE LOGS). Never leave
logs/README.md out of date.

### Rule 9: Three Error Categories — Check All Three
When auditing or validating, explicitly check for and report on all three:

1. **Assumption-level errors** — The most dangerous category. These are
   places where the system was built on something that was assumed to be
   true but was never verified. They pass all local checks and only fail
   at integration or runtime. Example from this project: the original
   assumption that Flutter generates session_id client-side. Every component
   looked correct in isolation; the system broke at the interface.
   Ask for every interface: "What did the builder assume the other side
   would provide? Is that assumption actually true?"

2. **Implementation bugs** — Code that is incorrect relative to the
   specification. Wrong field names, wrong logic, missing null checks,
   incorrect formula implementations.

3. **Other findings** — Anything else that represents risk: documentation
   gaps that will cause future developers (including future AI instances)
   to make wrong assumptions, naming inconsistencies that create confusion
   at boundaries, missing validation, deviation from stated best practices.

## RESOLUTION CYCLE

When findings are identified, the resolution follows this pattern:

1. Findings come to this chat (either via Claude Code Opus report pasted
   here, or via direct validation in this chat).
2. Architecture-level findings → this chat discusses with user → Architecture
   Chat also produces a solution → both solutions are compared before locking.
3. Backend/Frontend/Data findings → this chat generates a corrective prompt
   for the relevant specialist chat → specialist chat implements → this chat
   or Claude Code Opus validates the fix.
4. Cross-validation rule: when both this chat and a specialist chat have
   produced solutions for the same finding, both solutions are reviewed
   together. Either chat may catch something the other missed.

The goal of cross-validation is not redundancy — it is catching the gap
between "correct in isolation" and "correct as a system."

### AgentState — All 12 Fields
```python
from typing import Annotated, TypedDict
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    # ── Conversation ──────────────────────────────────────────────────
    messages: Annotated[list[BaseMessage], add_messages]
    # Full conversation history. add_messages reducer appends — never overwrites.

    # ── Student data ──────────────────────────────────────────────────
    student_profile: dict
    # Loaded from DB at session start using JWT sub. Never from request body.
    # Contains: riasec_scores, subject_marks, capability_scores,
    #           stream, education_level, grade_system.

    active_constraints: dict
    # Extracted by ProfilerNode conversationally.
    # Contains: budget_per_semester, transport_willing, home_zone,
    #           stated_preferences, family_constraints, career_goal, student_notes.
    # May be overridden per-session via context_overrides in ChatRequest.

    # ── Graph control ─────────────────────────────────────────────────
    profiling_complete: bool
    # True when budget_per_semester, transport_willing, home_zone all non-null.
    # For O/A Level students: stream confirmation also required.

    last_intent: str
    # Written by SupervisorNode every turn. Drives conditional edge routing.
    # Values: get_recommendation | profile_update | fee_query |
    #         market_query | follow_up | clarification | out_of_scope

    student_mode: str
    # "inter" | "matric_planning"
    # Set at session start from student_profile.education_level.

    education_level: str
    # Raw education_level string from student_profile.
    # Drives assessment question pool selection.

    # ── Roadmap data ──────────────────────────────────────────────────
    current_roadmap: list
    # Sorted list of scored degrees. Written by ScoringNode.

    previous_roadmap: list | None
    # Loaded at session start from most recent recommendations row.
    # None if no prior run. Used by ExplanationNode for "What Changed".

    thought_trace: list
    # Reasoning steps from FilterNode and ScoringNode. Shown in UI toggle.

    mismatch_notice: str | None
    # Set by ScoringNode when stated preference scores 20+ points below
    # top match AND preferred degree FutureValue < 6.

    conflict_detected: bool
    # MVP-3 placeholder — always False. Do not use in Sprint 1–4.
```

**Fields that must NOT be in AgentState:** `interests`, `onboarding_stage`
— these live in the database, not in AgentState.

### Agent Pipeline (in order)
1. SupervisorNode — LLM intent classification → Conditional Edges to specialist
2. FilterNode — three output lists: confirmed_eligible / likely_eligible / stretch
3. ScoringNode — total_score = (0.6 × match_score) + (0.4 × future_score),
   generates thought_trace JSONB
4. ExplanationNode — LLM, 3–4 sentence explanation per top recommendation
5. AnswerNode — handles fee_query / market_query / follow_up / clarification /
   out_of_scope (polite decline). Single node — replaces the planning-era
   UniversityAdvisorNode and MarketAnalystNode. If you see either of those
   names in code, flag it as an error.

### FilterNode Constraints (checked in this order)
1. Stream eligibility: `fully_eligible_streams` → confirmed |
   `conditionally_eligible_streams` → likely
2. Mandatory subjects: all items in `degree.mandatory_subjects` present in
   `student.subject_marks`
3. Marks: `student.aggregate >= degree.min_percentage_hssc`
4. Budget: `degree.fee_per_semester <= student.budget_per_semester`
5. Stretch: marks within 5% below `degree.cutoff_range.min`

**Hard exclusions (degree never appears):**
- Stream has no pathway to degree AND no conditional eligibility
- Mandatory subject missing AND no waiver pathway

**Everything else is a soft flag — never a hard exclusion.**
Minimum 5 degrees always shown regardless of marks.

### Stream Eligibility (2024 verified policy)
- Pre-Medical → Engineering: Conditionally eligible (PEC July 2024,
  8-week math bridge, max 40% seats)
- Pre-Medical → CS/Computing: Conditionally eligible (NCEAC 2020,
  1st year deficiency course)
- Pre-Engineering → MBBS/BDS: Hard blocked (PMDC 2023, permanent)
- NED Karachi engineering programs: `"policy_pending_verification": true`
  until official announcement
