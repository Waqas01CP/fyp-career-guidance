## WHY THIS WAS DONE

### Root cause — wrong design medium from the start

The HTML mockups were built using Tailwind CSS and web layout conventions.
Web and Flutter use fundamentally incompatible layout systems:

- Web: box model + CSS flexbox/grid + viewport units + hover states
- Flutter: widget tree + constraint propagation + logical pixels + InkWell

When Claude Code or Antigravity was given an HTML mockup and asked to
implement it in Flutter, it performed a translation between two incompatible
systems. This translation is inherently lossy — Flutter's constraint system
does not map cleanly to CSS properties.

This was the root cause of every iterative problem in the frontend:
- Keyboard overflow on login — fixed 5+ times across multiple sessions
- Black screens on navigation — PostFrameCallback timing mismatches
- Back-stack corruption — pushReplacementNamed vs pushNamed confusion
- Color inconsistencies — seedColor was AI purple instead of teal
- Spacing inconsistencies — CSS pixels translated literally to Flutter
- Shadow inconsistencies — hard CSS shadows became wrong Flutter BoxShadow
- Border radius inconsistencies — form cards 32px became 16px in Flutter

None of these were isolated implementation errors. They were all symptoms
of the same root cause: using web mockups as the Flutter design source.

### Secondary cause — AI-generated design without Flutter knowledge

The HTML mockups were generated by Claude Code, which does not have deep
native knowledge of Flutter's widget system. The mockups looked visually
reasonable as web pages but contained patterns that do not map to Flutter:
- CSS grid layouts (Flutter uses Row/Column/Wrap, not grid)
- Fixed pixel widths (Flutter uses flex/constraints, not pixel widths)
- Hover states (Flutter mobile has no hover — only InkWell/GestureDetector)
- Tailwind class names (completely incompatible with Flutter's naming)
- CSS variables as design tokens (Flutter uses ThemeData, not CSS vars)

The AI insight panel (purple) was documented as `#6616D7` in CSS hex but
not as `const Color(0xFF6616D7)` in Flutter — a minor difference that
accumulated into inconsistent implementations across 5+ screens.

### Why iterative improvement was insufficient

Each iteration fixed one surface symptom without fixing the root cause.
The root cause was always the same: the wrong design source for the medium.
No amount of patching HTML-translated Flutter code produces the quality
of Flutter code written from a Flutter-native specification from the start.

The 5-task refactor session (claude-code-2026-04-27-03-00-frontend-refactor)
fixed navigation, keyboard overflow, draft keys, and settings. It took
one full session to undo problems that originated in the HTML-to-Flutter
translation. With a Flutter-native specification from the start, that
session would not have been necessary.

## WHAT REPLACED THE DEPRECATED FILES

### docs/00_architecture/FRONTEND_SCREEN_CONTRACTS.md
Flutter-native specification for all 16 screens. Each screen defines:
- DATA IN: what it reads from backend or local state
- DATA OUT: what it writes to backend (endpoint + exact request body)
- STATES: every possible UI state (loading, loaded, error, empty)
- USER ACTIONS: what the student can do and what each action triggers
- NAVIGATION: exact Navigator call for each outcome
- STAGE GUARD: when and how to route forward past this screen
- DO NOT: screen-specific exclusions

Source of truth for "what each screen must do."
Visual decisions are NOT in this document — they are in FRONTEND_DESIGN_SYSTEM.md.

### docs/00_architecture/FRONTEND_DESIGN_SYSTEM.md
Flutter-native design system. Covers:
- ThemeData configuration (seed color, font, useMaterial3)
- Three-color role logic (locked, architecture-level)
- Material Design 3 color roles (use by role name, never by hex)
- Named color exceptions (AI purple system, amber warning, splash)
- Typography size reference (9px–30px specific use cases)
- Spacing (flutter_screenutil design canvas 390×844)
- Border radius (two values: form cards 32r, content cards 16r)
- Two shadow types (editorial and content)
- 16 component patterns with exact Flutter code
- Navigation rules
- Flutter-specific implementation rules
- Package name fix (com.example.frontend_app → com.fyp.career_guidance)
- Draft key pattern

Source of truth for "how each screen must look and behave visually."
